### 🟡 [A3-03] `reconciliation.ts` virou binário (bytes NUL crus) e escondeu a lógica central do diff — Tech Lead aprovou com leitura degradada; QA pegou

**Categoria:** Qualidade / Eficácia de gate
**Evidência:** O dev-front (HUB-92) usou **2 bytes NUL literais** (`\x00`) como separador no `groupKey` de `reconciliation.ts` (arquivo-núcleo do PR #38). Isso fez o git classificar o arquivo como **binário**: `git diff` → `Binary files … differ`, `numstat` `- -` (0/0). Consequência dupla: o arquivo central ficou **invisível no diff do PR**, e o Tech Lead — que revisou lendo o arquivo no worktree — ainda assim **descreveu o separador como "espaço"** (nit: *"reconciliation.groupKey usa separador espaço"*), ou seja, a lógica real (NUL) passou despercebida no gate de review. O **QA**, validando contra critérios e inspecionando os arquivos, identificou os NUL crus, explicou o impacto na reviewabilidade e **recomendou exigir correção antes do merge**. O dev trocou por escape `\0` (runtime idêntico), re-rodou o gate (lint 0 err · `tsc` limpo · 395 testes) e o arquivo voltou a `numstat 100 0`.
**Impacto:** Positivo líquido — **a rede de gates funcionou**: a camada QA capturou o que a camada Tech Lead não viu, e nada defeituoso foi mergeado. Mas expõe uma lacuna: um PR cujo arquivo-núcleo é invisível no diff **chegou à aprovação do Tech Lead**. Sem a captura do QA, teria mergeado sem review real da lógica.
**Sugestão:** Guard automático barato: no gate de PR, falhar/alertar quando um arquivo de **código-fonte** (`.ts/.tsx/.js/.py`) for classificado como binário (`git diff --numstat` com `- -`) ou contiver bytes de controle não-imprimíveis (`grep -aP '[\x00-\x08\x0e-\x1f]'`). Orientar devs (regra de Clean Code): sem bytes de controle crus embutidos — usar escapes.
**Responsável sugerido:** Tech Lead (regra + check) + Orchestrator (adicionar ao gate)
**Urgência:** Próxima sprint

### 🟢 [A3-06] `cd … &&` cresceu para 183 ocorrências (de 72) — erradicar o anti-padrão sandbox

**Categoria:** Conformidade / Otimização
**Evidência:** 183 de 352 comandos `Bash` (52%) usam `cd <path> && …`, contra apenas 16 `git -C` e 33 invocações por caminho absoluto/`--prefix`. 0 aprovações reais foram disparadas (sorte de config), mas 5 agentes auto-atestaram "nenhum comando exigiu aprovação" usando o padrão proibido.
**Impacto potencial:** Risco latente de estol por aprovação manual (o `cd … &&` é exatamente o que a regra veta); erradicá-lo remove o risco e alinha DoD ↔ prática. Ganho de robustez, não de tempo imediato.
**Sugestão:** Substituir por `git -C <path> …`, `npm --prefix <path> run <script>` e binários por caminho absoluto (`<worktree>/node_modules/.bin/vitest`). Adicionar 2–3 exemplos concretos nos prompts de dev-front/dev-back/tech-lead/qa e um lembrete no `.claude/agents/`.
**Responsável sugerido:** Tech Lead (padrão) + Orchestrator (prompts)
**Urgência:** Backlog de melhoria

## Padrões identificados

- **A engenharia amadureceu; a cerimônia regrediu.** SDD, AUD-04, ordenação por dependência, worktrees e gates estão sólidos e auditáveis. Os furos são de **ritual de alinhamento** (PO/PM Sync ausente de novo; Daily retroativo; modelo de sprint ambíguo) — a mesma classe de achados de 24/06 e 30/06. **Sistêmico:** recomendações de cerimônia da Retro anterior não foram efetivadas.
- **Rede de gates multicamada funciona.** O caso `reconciliation.ts` é a prova: uma camada (Tech Lead) falha parcialmente, a seguinte (QA) captura. Vale preservar e reforçar com um guard automático barato.
- **Recorte de spec funciona — mas só onde foi aplicado.** O ganho de eficiência é real e visível; o risco é depender de disciplina ad-hoc do Orchestrator em vez de regra codificada — a próxima tarefa grande repetirá o 302k se nada mudar.
- **`cd … &&` persistente e crescente** (72 → 183) apesar de regra explícita e do DoD: diretriz não internalizada nos prompts dos agentes — reincidência do tema AUD-01/PA-04.
